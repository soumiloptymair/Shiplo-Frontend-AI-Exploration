Feature: Merge/Split Order Functionality
2. Overview
This feature introduces tools within the Order Management Module that empower users to:
Split Orders: Divide one order into two or more shipments, each with its own tracking and delivery details.
Merge Orders: Combine multiple orders (with compatible delivery and customer details) into a single shipment for easier management and lower cost.
The system ensures data integrity, inventory synchronization, and accurate cost allocation between the merged/split shipments.
1. Business Functionality
The Merge/Split Order Functionality allows shippers to manage complex shipment scenarios by combining multiple orders into one shipment or dividing a single order into multiple shipments. This enhances operational flexibility, optimizes shipping costs, and supports multi-location deliveries or order consolidation for customers.
Business Objectives
Improve shipment efficiency and cost-effectiveness.
Allow customers to fulfill orders to multiple addresses or destinations.
Enable consolidation of multiple small orders for bulk shipment savings.
Reduce manual handling of shipment grouping and splitting.

7. Assumptions and Constraints
Assumptions
HEALE API handles all label generation and recalculated shipping costs.
Payment settlements for additional shipping costs are processed via HEALE Wallet.
User has appropriate permissions for modifying shipment structures.
Constraints
Orders with different carriers or destinations cannot be merged.
Splitting orders is allowed only before the label is generated.
Merge/Split operations are not reversible after label printing.
Product inventory levels at the fulfillment center must be sufficient to fulfill order(s).

Merge/Split Recommendations Engine
Hard Constraints (HC): Must be true; otherwise the engine must not recommend.
 (H): If true, increase a candidate’s score.
Penalties (P): If true, decrease a candidate’s score.
Thresholds (T): Default values you can tweak per account, lane, or carrier.
The engine outputs:
Action: MERGE (specific order set) or SPLIT (specific order + split plan).
Score: 0–100 (confidence).
Rationale: Brief, human-readable reasons + key fields used.
Global Hard Constraints (apply to both merge and split)
Label lockout (HC): If a shipment label is already purchased/manifested or an order is customs-locked (IOSS/HS code finalized), do not recommend changes unless reversible without fees.
Regulatory compatibility (HC): Do not recommend if combining would mix hazmat/non-hazmat, age-restricted items, or temperature-control requirements.
Carrier/service incompatibility (HC): Orders using incompatible carriers or service levels (e.g., Ground vs. 2-Day Air) must not be merged; split parts must each map to a valid service.
Destination rules (HC): Never merge across different final destinations or customs jurisdictions that would require separate declarations.
Payment status (HC): Orders with pending/failed payment, KYC flags, or fraud holds are excluded from merge; split only allowed if it isolates the blocked line items.
SLA integrity (HC): Changes must not violate promised delivery dates/cut-offs already shown to the customer.
Merge Recommendation Rules
Candidate Pooling
Build candidate sets among open, unmanifested orders sharing destination_address within the consolidation window (T1).
T1 (default): 6 hours from firs+t qualifying order, capped by carrier pickup cut-off.
Hard Constraints (Merge-specific)
Same ship-to address (HC): Exact match, including apartment/suite; only if verified via address normalization.
Service alignment (HC): Same required delivery date or compatible services that meet the earliest SLA.
Capacity limits (HC): Sum of dims/weight must fit within one parcel/pallet under chosen service (consider dimensional weight and carrier maxes).
Scoring Heuristics
For each candidate set C:
H1 Cost savings (weight 30): Estimated shipping cost reduction ≥ T2 yields linear score boost.
T2: 10% savings minimum to recommend; +1 point per additional 2% up to 30.
H2 Package utilization (15): Combined dims fit a package preset with utilization 70–90% (optimal) → +15; 50–70% or 90–95% → +8; else +0.
H3 SLA alignment buffer (10): All orders in C meet the earliest promised date with ≥ T3 slack → +10; half slack → +5.
T3: 12 hours.
H4 Handling similarity (10): Same fragility, temperature, insurance needs → +10; minor differences → +5.
H5 Warehouse wave efficiency (10): Orders in C pick from the same zone/bin wave within T4 → +10.
T4: 2 hours picking window.
H6 Returns simplification (5): Same returns address/RMA policy → +5.
Penalties
P1 Delay penalty (−20 to −5): If merging requires holding any order beyond T6 (but still meeting SLA), subtract up to 20 based on hold time.
T6: 2 hours soft hold, 6 hours hard cap.
P2 Risk mix (−10): Mixing high-value and low-value items with theft-risk uplift.
P3 Carrier surcharge risk (−10): Likely to incur oversize/additional handling.
P4 Packaging constraint ambiguity (−10): No preset/package confidently fits.
Recommendation Thresholds
Recommend MERGE if total score ≥ 65 and no penalties push SLA risk > 5% modeled probability.
Auto-apply (optional policy): if score ≥ 85, cost savings ≥ 20%, and no P-class penalties.
Split Recommendation Rules
Split Triggers
Evaluate a single order O for split when any of the following true:
Hard Constraints enabling split
Multi-destination (HC): Explicitly assigned to different addresses or recipients.
Backorder/availability (HC): At least one product is not available within T7 at the fulfillment center/warehouse; others are ready.
T7: 24 hours.
Service incompatibility (HC): Requires different service (e.g., hazmat ground vs. air, temperature control, oversize).
Carrier limits (HC): Combined dims/weight exceed single-package limits for chosen service.
Split Plan Heuristics
For each feasible split plan S = {S1…Sn}:
H1 SLA protection (25): Delivery meeting promised date only if shipped now → +25.
H2 Cost tradeoff (20): Incremental shipping cost per additional package ≤ T8 and/or surcharge avoided ≥ T9.
T8: +$6 per extra package; T9: avoiding ≥ $20 oversize/ADH surcharge.
H3 Customer experience (10): Splitting reduces delivery lead time variance (p90–p10) below T10 → +10.
T10: 2 days.
H4 Packaging fit (10): Each split fits optimal presets (utilization 70–90%).
H5 Inventory locality (10): Ship available products from nearest warehouse/fulfillment center to customer; backordered from alternate warehouse/fulfillment center with ETA.
H6 Risk isolation (5): Separate fragile/high-value or signature-required items.
Penalties
P1 Extra touchpoints (−15): Additional packages beyond T11 → subtract per extra package.
T11: 2 packages included; −5 for each beyond 2.
P2 Customer annoyance (−10): If marketing/checkout promised “one delivery,” apply penalty unless customer opted into split-shipping.
P3 Returns complexity (−5): If split creates multiple RMA flows for the same order.
Recommendation Thresholds
Recommend SPLIT if plan score ≥ 60 and:


Any line would otherwise miss SLA, or
Splitting avoids oversize/surcharge ≥ $20, or
Splitting isolates restricted items (hazmat, temp-control).


Preferred Split Pattern Order:


Ready-to-ship now vs. backordered later
Different service requirements (e.g., cold chain vs. standard)
Oversize isolation (one big item separate)
Fragile/value isolation
Conflict Resolution (Merge vs. Split)
If both a merge candidate and a split plan exist for overlapping items:
SLA first: Choose the option that maximizes % delivery meeting promised date (break ties by higher score).
Cost next: Compare total landed cost (labels + materials + handling).
CX policy: Honor storefront promise (single delivery vs. fastest delivery).
Tiebreaker: Prefer the option with lower modeled risk (damage/theft).
Operational Guardrails
Cut-off awareness: Never recommend actions that cross carrier pickup cut-off without re-estimating SLA.
Undo window: Only auto-apply actions if still within undo window (e.g., 10 minutes) and no pick has started.
Picker state check: If any carton is scanned as “packed,” lock that carton from merge; allow split only of the remaining order.
Explainability: Always return the top 3 reasons driving the score and which rule(s) gated the decision.
A/B policy flags: Allow per-merchant toggles: “cost-first,” “SLA-first,” “CX-first,” which adjust weights by ±10%.
Data Inputs Needed
Order: SKU, qty, dims/weight, handling tags (fragile, hazmat, temp-control), value, signature requirement.
Address normalization & geocode; customer ID.
Inventory by FC with ATS and ETA.
Carrier matrix: limits, surcharges, cut-offs, service SLAs, dimensional weight rules.
Package presets & materials cost.
Promised delivery date per order (from checkout).
Risk and emissions models (or default heuristics).
Scoring Pseudocode (illustrative)
if violates_any(HC_global or HC_action_specific): return REJECT

score = 0
for h in applicable_heuristics: score += h.weight * h.value   # normalized 0–1
for p in applicable_penalties: score -= p.weight * p.value
if action == MERGE and score >= 65: recommend
if action == SPLIT and score >= 60: recommend
